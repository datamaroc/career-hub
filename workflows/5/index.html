قبل البدء، يجب عليك استيراد ملف rag.json المرفق إلى بيئة n8n الخاصة بك. هذا الملف يحتوي على الهيكل الكامل لسير العمل وجميع العقد (Nodes) مترابطة وجاهزة للتكوين.

  1. في لوحة تحكم n8n، اذهب إلى قسم Workflows.

فهم المفاهيم الأساسية (Key Concepts)

🤖 وكيل الذكاء الاصطناعي AI Agent

›

من منظور علم البيانات، تخيل النموذج اللغوي (LLM) كعالم فيزياء عبقري، لكنه محبوس في غرفة فارغة بذاكرة لا تتجاوز 5 ثوانٍ. يمكنه حل أي معادلة معقدة ترميها عليه، لكنه لا يعرف من أنت ولا يستطيع طلب بيتزا.

الـ AI Agent هو نفس هذا العالم، لكننا قمنا بترقيته! أعطيناه:

  • دفتر ملاحظات (Memory)
  • صندوق أدوات سويسري (Tools)
  • اتصال بالعالم الخارجي (APIs)

فجأة، أصبح بإمكانه تذكر محادثاتكم السابقة، البحث عن أفضل مطعم بيتزا، وطلبها نيابة عنك. إنه لم يعد مجرد "متحدث"، بل أصبح "فاعل".

المكونات الأساسية للوكيل:

  • 🧠
    نموذج الدردشة (Chat Model):

    هذا هو "المحرك" أو العقل المفكر (LLM). هو الذي يقوم بعملية التفكير، التخطيط، واتخاذ القرار. يحلل طلبك ويقرر: "هل يجب أن أجيب مباشرة، أم أستخدم إحدى أدواتي؟". بدون هذا العقل، الوكيل مجرد هيكل بلا روح.

  • 💾
    الذاكرة (Memory):

    حتى أذكى العقول تحتاج إلى ذاكرة. وهي تأتي بنكهتين:

    • ذاكرة قصيرة المدى: تتذكر آخر أجزاء المحادثة (مثلاً، آخر 5 رسائل) حتى لا يسألك الوكيل "عن ماذا كنا نتحدث؟" كل دقيقة.
    • ذاكرة طويلة المدى: هذه هي مكتبته المعرفية، أي قاعدة البيانات الفيكتورية (VectorDB) التي يمكنه البحث فيها عن حقائق ومعلومات تعلمها سابقًا.
  • 🛠️
    الأدوات (Tools):

    هذه هي "أذرع وأعين" الوكيل. هي وظائف (Functions) محددة يمكن للعقل (النموذج) أن يقرر استدعاءها. في سير العمل هذا، الأداة الرئيسية هي أداة البحث في VectorDB. يمكن أن تكون هناك أدوات أخرى مثل البحث في جوجل، الوصول للآلة الحاسبة، أو استدعاء API الطقس.

نظرة تقنية على حلقة التفكير (ReAct Loop):

يعمل الوكيل في حلقة مستمرة من التفكير والفعل والملاحظة، وهو إطار عمل يُعرف باسم ReAct (Reasoning and Acting).

// User asks: "What is the main idea in the 'Q3-Strategy.pdf' document?"

Thought: "The user is asking about a specific document. I cannot answer from my general knowledge. I need to use my document search tool."
Action: search_vector_db(query="main idea of Q3-Strategy.pdf")

// Tool returns a result
Observation: "Retrieved chunk: 'The core of our Q3 strategy is to focus on expanding into the European market...'"

Thought: "I have found the relevant information. I can now formulate a final answer based on this observation."
Final Answer: "The main idea in the 'Q3-Strategy.pdf' document is to focus on expanding into the European market."

🧠 تقنية RAG

›

النماذج اللغوية الكبيرة (LLMs) تشبه المؤرخين العباقرة الذين قرأوا كل كتاب تم نشره حتى عام 2023. إنهم أذكياء بشكل لا يصدق، لكن لديهم مشكلتان كبيرتان:

  • معلوماتهم قديمة: إذا سألتهم عن أخبار الأمس، سيقولون "لا أعرف".
  • يمكن أن "يهلوسوا": إذا لم يعرفوا الإجابة، قد يخترعون واحدة بثقة تامة!

تقنية RAG (Retrieval-Augmented Generation) هي الحل. إنها ببساطة مثل أن تعطي المؤرخ "جريدة اليوم" (معلومات من VectorDB) قبل أن تطرح عليه سؤالك، وتطلب منه أن يبني إجابته على ما جاء في الجريدة فقط.

العملية من خطوتين:

  • الاسترداد (Retrieval): عندما تطرح سؤالاً، يقوم النظام بالبحث في VectorDB عن أكثر المستندات صلة بسؤالك. هذه هي "قصاصات الجريدة" التي سنسلمها للمؤرخ.
  • التعزيز والتوليد (Augmented Generation): هذه هي الخطوة السحرية. بدلاً من إرسال سؤالك فقط إلى النموذج، نقوم ببناء "موجه معزز" (Augmented Prompt) يتضمن السياق الذي وجدناه.

مثال تقني: الفرق بين الموجه العادي والمعزز

// User asks: "What are the key risks for Project Alpha?"

// -- WITHOUT RAG --
// The LLM gets a simple, context-free prompt. It might hallucinate.
prompt_without_rag = "What are the key risks for Project Alpha?";

// -- WITH RAG --
// 1. First, we retrieve relevant context from our VectorDB.
retrieved_context = "From meeting_notes.pdf: Project Alpha's primary risk is supply chain disruption from our main vendor in Asia. A secondary risk is the upcoming regulatory change in Q4."

// 2. We build a new, much better prompt.
augmented_prompt = ` Based ONLY on the following context, answer the user's question. Context: """ ${retrieved_context} """ User Question: "What are the key risks for Project Alpha?" `;
// Now, the LLM has everything it needs to give a factual, non-hallucinated answer. It's a whole different game!

📚 قاعدة بيانات VectorDB

›

تخيل أن لديك مكتبة تقليدية. إذا طلبت من أمين المكتبة كتابًا عن "إصلاح المركبات"، ولكنه لا يجد هذا العنوان بالضبط، فسيعود خالي الوفاض، حتى لو كان لديه كتاب بعنوان "تصليح السيارات". هذا هو البحث بالكلمات الرئيسية، وهو محدود جدًا.

قاعدة البيانات الفيكتورية (VectorDB) هي مثل أمين مكتبة فائق الذكاء. إنه لا يبحث عن الكلمات، بل يفهم "المعنى". عندما تطلب منه "إصلاح المركبات"، فإنه يفهم أنك تقصد نفس الشيء مثل "تصليح السيارات" ويحضر لك الكتاب الصحيح.

كيف تعمل؟ عن طريق "العناوين الدلالية"

العملية بسيطة في مفهومها:

  • التضمين (Embedding): كل قطعة نص (مستند، فقرة) يتم تحويلها إلى سلسلة من الأرقام تسمى "متجه" (vector). هذا المتجه هو بمثابة "عنوان دلالي" فريد للنص، يحدد موقعه في فضاء المعاني.
  • البحث بالتشابه: عندما تطرح سؤالاً، يتم تحويله هو الآخر إلى "عنوان دلالي". ثم تطلب قاعدة البيانات من أمين المكتبة الذكي: "ابحث لي عن العناوين الأقرب لهذا العنوان الجديد".

مثال تقني:

// 1. We have a document.
doc_text = "Our Q3 revenue showed a 15% growth, driven by the new European market.";

// 2. We convert it to a "semantic address" (a vector).
doc_vector = embedding_model.embed(doc_text);
vector_db.add(doc_vector, doc_text);

// 3. A user asks a question with different words but similar meaning.
user_question = "How did our finances do last quarter in Europe?";
question_vector = embedding_model.embed(user_question);

// 4. Find the document with the closest address.
results = vector_db.find_nearest(question_vector);
// The result is our original document. Simple, powerful, and no keyword fuss.

الخطوة 2: إعداد المتطلبات وبيانات الاعتماد (Prerequisites & Credentials)

1. إعداد Supabase (قاعدة البيانات والذاكرة)

›

أ. تهيئة قاعدة البيانات (تنفيذ أمر SQL)

هذه خطوة أساسية لإنشاء الجداول والوظائف اللازمة في قاعدة بياناتك. يجب تنفيذها مرة واحدة فقط لكل مشروع.

  1. اذهب إلى لوحة تحكم مشروعك في Supabase.
  2. من القائمة الجانبية، اختر SQL Editor (أيقونة تشبه ورقة بها علامة </>).
  3. اضغط على + New query.
  4. انسخ الكود التالي بالكامل والصقه في المحرر.
    5. -- Enable the pgvector extension
create extension vector;
-- Create a table for your documents
create table documents (
  id bigserial primary key,
  content text,
  metadata jsonb,
  embedding vector(1536)
);
-- Create a function to search for documents
create function match_documents(
    query_embedding vector(1536),
    match_count int,
    filter jsonb DEFAULT '{}'
)
returns table (
    id bigint,
    content text,
    metadata jsonb,
    similarity float
)
language plpgsql
as $$
begin
  return query
  select
    id,
    content,
    metadata,
    1 - (documents.embedding <=> query_embedding) as similarity
  from documents
  where metadata @> filter
  order by documents.embedding <=> query_embedding
  limit match_count;
end;
$$;
  5. اضغط على الزر الأخضر الكبير RUN في أسفل يمين المحرر.

ب. استخراج بيانات الاعتماد (Credentials)

ستحتاج إلى مجموعتين من بيانات الاعتماد من Supabase:

  • بيانات API (لعقد Vector Store):
    1. اذهب إلى Project Settings (أيقونة الترس ⚙️).
    2. اختر قسم API.
    3. ستجد Project URL و Project API Keys. انسخ الـ URL و مفتاح anon public.
  • بيانات PostgreSQL (لعقدة Chat Memory):
    1. اذهب إلى لوحة تحكم مشروعك في Supabase.
    2. في الشريط العلوي (navbar)، اضغط على زر Connect.
    3. ستظهر نافذة منبثقة بعنوان Connect to your project.
    4. مرر للأسفل حتى تصل إلى قسم Transaction pooler.
    5. اضغط على View parameters. ستظهر لك جميع تفاصيل الاتصال التي تحتاجها (Host, Database, User, Port, Password).

2. إعداد OpenAI (محرك الذكاء)

›

تحتاج فقط إلى مفتاح API واحد لاستخدامه في جميع عقد OpenAI.

  1. سجل الدخول إلى حسابك في platform.openai.com.
  2. من القائمة الجانبية، اذهب إلى API Keys.
  3. اضغط على + Create new secret key.
  4. أعط المفتاح اسمًا (مثلاً n8n-RAG-Key) واضغط Create secret key.
  5. انسخ المفتاح فورًا واحفظه في مكان آمن. لن تتمكن من رؤيته مرة أخرى.
